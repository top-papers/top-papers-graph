# Пайплайн качества: архитектура Propose-Verify + оценка качества утверждений

## Обзор

Пайплайн качества превращает зашумлённое LLM-извлечение триплетов в надёжную генерацию научных гипотез. Решает фундаментальную проблему: малые LLM (7B параметров) порождают ~50% шума при извлечении триплетов из научных статей.

## Архитектура

```
Статьи (PDF) → Чанки (1200 символов)
                    ↓
    Слой 1: Закалённый промпт («до 5», негативные примеры)
                    ↓  ~2 триплета/чанк (было 3-10)
    Слой 2: LLM-верификация (пакетная проверка: valid/noise/speculative)
                    ↓  ~80% проходят (21% отсев)
    Слой 3: Оценочная функция (10 признаков → взвешенная сумма → sigmoid → [0,1])
                    ↓  ранжирование по качеству
    Выход: Верифицированный каузальный граф (~131 ребро/домен, 70% каузальных)
```

## Слой 1: Закалённый промпт

**Файл:** `src/scireason/temporal/temporal_triplet_extractor.py`

Изменения по сравнению с базовой версией:
- Снижена квота извлечения: «3-10» → «до 5» с явным «лучше 1-2 точных, чем 5 сомнительных»
- Добавлены 3 негативных примера: что НЕ извлекать (структура статьи, расплывчатые сущности, формулы)
- Закрытый словарь предикатов: 19 канонических предикатов перечислены явно
- Запрещён `associated_with` (75% отсев на бэктесте)
- Руководство по сущностям: «конкретные доменные термины, 2-5 слов»

## Слой 2: LLM-верификация

**Файл:** `src/scireason/temporal/triplet_verifier.py`

Пост-экстракционный фильтр, работающий на собранном словаре рёбер:
1. **Порог confidence:** удаляет рёбра с `mean_confidence < threshold`
2. **Пакетная LLM-проверка:** рёбра группируются по статье и отправляются в LLM для классификации
3. **Fallback:** при отказе LLM — правила на основе типа предиката и повторяемости

Классификация: `valid` (оставить) / `noise` (удалить) / `speculative` (оставить с пониженным приоритетом)

## Слой 3: Оценочная функция качества утверждений

**Файл:** `src/scireason/temporal/assertion_scorer.py`

Взвешенная оценочная функция (weighted scoring function), заменяющая ненадёжную LLM-confidence (почти всегда ≈ 1.0) на вычисляемую метрику качества.

### Формула

```
score(edge) = sigmoid(w · features + bias) ∈ [0, 1]
```

Это взвешенная сумма 10 нормализованных признаков с sigmoid-нормализацией в [0, 1]. **Логистическая регрессия** используется исключительно как аналитический инструмент для подбора весов `w` и `bias` на размеченных данных — вместо того, чтобы оставлять их экспертными.

### 10 признаков

| # | Признак | Что измеряет |
|---|---------|-------------|
| 0 | redundancy | Сколько чанков независимо извлекли это ребро |
| 1 | entity_specificity | IDF сущностей (редкие > частые) |
| 2 | predicate_strength | Экспертная таблица: causes=1.0, is_a=0.2 |
| 3 | mean_confidence | LLM self-reported confidence (слабый сигнал, вес занижен) |
| 4 | evidence_grounding | Токенное пересечение цитаты и сущностей |
| 5 | evidence_count | Количество подтверждающих цитат |
| 6 | paper_diversity | Поддержка из нескольких статей |
| 7 | polarity_clarity | Согласованность полярности (supports/contradicts) |
| 8 | graph_integration | Связанность вершин ребра с остальным графом |
| 9 | pmi_normalized | Pointwise mutual information (информативность) |

### Подбор весов

Комбинированный лосс для градиентного спуска:
```
L = α·L_rank + β·L_bce + γ·L_reg
```

- **L_rank:** попарная margin ranking loss (pos vs neg) — устойчива при малом числе негативов
- **L_bce:** бинарная кросс-энтропия для абсолютной калибровки
- **L_reg:** L2-регуляризация

Данные для обучения: вердикты LLM-верификации (принятые рёбра = positive, отклонённые = negative).

## Автоматический triage + HTML-клиент

**Файл:** `src/scireason/task2_triage.py`

Эвристическая классификация каждого триплета в accept/reject/review:
- Детекция шумовых сущностей (формулы, числа, стоп-слова, общие термины)
- Оценка качества предиката
- Интеграция score оценочной функции

Интерактивный HTML-клиент с горячими клавишами (A/R/U/E), фильтрацией и экспортом.

## Независимая оценка (LLM-as-Judge)

**Файл:** `src/scireason/eval/critic_agent.py`

Для разрыва циклической валидации (скорер обучается на авто-метках, тестируется на авто-метках) в проекте реализована независимая adversarial-оценка:

1. **Агент-критик** — отдельный LLM-вызов (ollama, промпт «скептический рецензент»), который независимо классифицирует каждый триплет как valid/noise/borderline.
2. **Тестовый набор** — экспертные рёбра из 20+ траекторий + 15 синтетических шумовых триплетов.
3. **Метрики** — precision, recall, F1, Cohen's kappa по меткам критика.

### Результаты (qwen2.5:7b-instruct, 55 триплетов)

Критик **значительно строже** экспертной разметки: он классифицирует 60% экспертных рёбер как «borderline» и 30% как «noise», принимая лишь 10% как «valid».

**Причина:** экспертные траектории описывают **нарративные каузальные цепочки** между статьями («FedAvg leads_to FedProx»), а критик ожидает **конкретные проверяемые результаты** с количественной поддержкой. Это разные уровни абстракции.

**Детекция шума:** precision=0.80 — критик почти не пропускает мусор как валидный.

## Конфигурация

```bash
# Включить все улучшения:
QUALITY_PRESET=all ./run_task2.sh --llm

# Отдельные переключатели:
TRIPLET_PAPER_CONTEXT_ENABLED=true          # Контекст статьи в промпте
ENTITY_NORMALIZATION_ENABLED=true           # Fuzzy-нормализация сущностей
PAPER_SUMMARY_TRIPLETS_ENABLED=true         # Summary-триплеты
CANONICAL_VOCABULARY_ENABLED=true           # Каноническая лексика
TRIPLET_VERIFY_ENABLED=true                 # LLM-верификация
ASSERTION_SCORER_ENABLED=true               # Оценочная функция (по умолчанию: вкл.)
```

```bash
# Запустить triage на bundle:

# Подобрать веса оценочной функции:

# Запустить независимую оценку (LLM-критик):
python scripts/eval/run_critic_evaluation.py
```

## Как получить веса оценочной функции

Файл весов (`data/derived/assertion_scorer_weights.json`) **не хранится в репозитории** —
он генерируется из размеченных данных. Порядок получения:

```bash
# 1. Подготовить Task 2 bundle (извлечь триплеты из статей):
scireason prepare-task2-validation --trajectory <path_to_trajectory.yaml>

# 2. Запустить triage (авторазметка accept/reject/review):
scireason triage-triplets --bundle-dir runs/task2_validation

# 3. (опционально) Скорректировать разметку через HTML-клиент,
#    экспортировать как triage_verdicts.json

# 4. Обучить веса:
scireason train-scorer --bundle-dir runs/task2_validation \
    [--labels triage_verdicts.json] \
    --out data/derived/assertion_scorer_weights.json

# 5. Готово. При отсутствии файла весов система использует
#    экспертные DEFAULT_WEIGHTS (заданы в assertion_scorer.py).
```

| Evidence grounding | частичный | 99% |

## Известные ограничения

1. **28% неточных формулировок** в топ-утверждениях (расплывчатые сущности, неточные предикаты)
2. **Нет кросс-документного синтеза** — статьи анализируются изолированно
3. **Низкий recall** по дизайну («до 5 триплетов на чанк»)
4. **Тестировано только на ML/CS статьях**
5. **Оценочная функция ловит только поверхностные признаки** — не заменяет LLM-верификацию для пограничных случаев
6. **Разрыв между уровнями абстракции** — экспертные траектории описывают нарративные каузальные цепочки, а скорер обучен на finding-level триплетах
