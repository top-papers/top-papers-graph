# Шпаргалка для ведущего демо-сессии

## Что это за блокнот
Этот notebook показывает две роли в Task 3 проекта `top-papers-graph`:
1. **Эксперт-создатель теста** — готовит тестовый набор и case manifest.
2. **Эксперт-участник blind A/B** — получает готовую слепую форму и заполняет review JSON.

Демо построено на реальных файлах репозитория и использует ту же логику, что и Task 3 offline review pipeline: trajectory/YAML-артефакты из Task 1, case manifests и blind review bundle для A/B оценки. В публичном репозитории есть основной README, обновлённый README по ноутбукам, notebook для Task 1 trajectory forms и Task 3 blind-review notebooks, так что сценарий демо совпадает с текущей структурой проекта. См. также:
- `README.md`
- `UPDATED_NOTEBOOKS_README.md`
- `task1_reasoning_trajectories_onine_offline_forms.ipynb`
- `task3_dual_local_models_blind_ab.ipynb`
- `task3_multimodal_temporal_hypothesis_generation.ipynb`

## Цель демо-сессии
Показать двумя последовательными частями, как в Task 3:
- сначала **создаётся корректный тестовый набор** для оценки различий между baseline VLM и tuned VLM;
- затем **другой эксперт независимо заполняет blind A/B review**, не зная, где baseline, а где tuned model.

Главная мысль для аудитории:
**Task 3 оценивает не просто “красоту финальной гипотезы”, а качество multimodal evidence extraction, evidence linkage и temporal grounding.** Поэтому создатель теста и участник теста выполняют разные, но взаимосвязанные функции.

## Как объяснить, зачем это проекту
> Так команда проверяет, действительно ли дообученный VLM лучше помогает всему Task 3 pipeline, а не просто формулирует текст чуть иначе.

# 7. Частые вопросы и быстрые ответы

## Вопрос: почему нельзя просто сравнить финальные hypotheses?
Потому что downstream text model может сгладить различия. Для Task 3 важнее увидеть, кто лучше извлекает и связывает evidence.

## Вопрос: почему создатель и участник — это разные люди?
Чтобы blind A/B оставался честным: участник не должен знать внутреннюю логику составления пар и соответствие вариантов моделям.

## Вопрос: почему есть `easy_control` кейсы?
Чтобы проверить, что tuned model не “побеждает всегда”, а выигрывает именно на сложных multimodal/temporal кейсах.

## Вопрос: почему в демо используются synthetic outputs?
Чтобы показать структуру процесса быстро. Полный запуск моделей для обучающей сессии слишком длинный.
