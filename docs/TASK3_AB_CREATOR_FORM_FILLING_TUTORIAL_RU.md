# Пошаговый туториал по заполнению офлайн-формы Task 3 для эксперта-создателя

Этот документ относится к обновлённой русской офлайн-форме для эксперта-создателя набора A/B теста в Task 3.

## Что изменилось в форме

Теперь в форме есть:
- кнопка **«Скачать черновик JSON»**;
- кнопка **«Загрузить черновик JSON»**;
- кнопка **«Очистить локальный черновик»**;
- отдельная кнопка **«Открыть Google Форму для отправки данных»**;
- автосохранение в браузере;
- понятные русские подсказки под полями.

## Когда использовать какой способ сохранения

### Автосохранение в браузере
Подходит, если:
- вы работаете в одном и том же браузере;
- не планируете переносить черновик на другой компьютер;
- хотите просто не потерять изменения при случайном обновлении страницы.

### Скачать черновик JSON
Используйте, если:
- хотите продолжить заполнение позже;
- работаете в несколько сессий;
- хотите передать незавершённый черновик коллеге;
- хотите иметь резервную копию.

### Скачать заполненный JSON
Используйте, когда:
- кейсы уже проверены;
- статусы готовности в форме не показывают критичных пробелов;
- вы готовы передать manifest в runner или в следующий этап процесса.

### Google Форма
Используйте, если:
- в вашей команде данные собираются централизованно через веб-форму;
- нужно передать результаты в единый канал сбора;
- требуется ручная отправка после локальной подготовки.

Ссылка:
`https://forms.gle/h5RwEA8DsZh9pBAt8`

## Пошаговый сценарий работы

## Шаг 1. Откройте форму
Откройте файл:
`task3_ab_creator_offline_form_ru.html`

После открытия вы увидите:
- верхний блок с описанием назначения формы;
- панель быстрых действий;
- блок метаданных;
- список кейсов.

## Шаг 2. Сначала заполните метаданные
В блоке **«Метаданные эксперимента»** заполните:
- `topic`
- `submission_id`
- `creator_id`
- `cutoff_year`
- `review_goal`

### Как заполнять
- `topic` — тема тестового набора.
- `submission_id` — идентификатор эксперимента.
- `creator_id` — имя или ID эксперта-создателя.
- `cutoff_year` — верхняя граница года публикаций.
- `review_goal` — короткое объяснение, что должен показать A/B тест.

### Хороший пример `review_goal`
> Проверить, где tuned VLM лучше baseline извлекает visual evidence из figures/tables и точнее сохраняет temporal grounding.

---

## Шаг 3. Добавьте кейсы по нужным стратам
Используйте кнопки:
- **Добавить multimodal_hard**
- **Добавить temporal_hard**
- **Добавить easy_control**

### Рекомендуемый порядок
1. Сначала добавить основные диагностические кейсы `multimodal_hard`.
2. Потом добавить `temporal_hard`.
3. В конце добавить немного `easy_control`.

### Рекомендуемая пропорция
- основной упор на `multimodal_hard`;
- затем `temporal_hard`;
- `easy_control` — в меньшинстве.

---

## Шаг 4. Заполните блок «A. Контекст кейса»
Для каждого кейса заполните:
- `case_id`
- `paper_title`
- `paper_id`
- `year`
- `evidence_kind`
- `page_hint`
- `stratum`

### Практический смысл
Этот блок помогает понять:
- к какой статье относится кейс;
- где именно сидит ключевой факт;
- какой тип сложности у кейса.

### Хорошие примеры
- `paper_id`: `doi:10.1000/example`
- `page_hint`: `p.4, Fig.2` или `Table 3`
- `evidence_kind`: `figure`, `table`, `figure_or_table`

---

## Шаг 5. Заполните блок «B. Что должен оценивать участник A/B»
Здесь два самых важных поля:
- `creator_prompt`
- `creator_rationale`

### Как писать `creator_prompt`
Это короткая инструкция для эксперта-участника.

Хороший prompt:
- короткий;
- конкретный;
- про одну главную задачу;
- понятный без технических пояснений.

### Пример хорошего `creator_prompt`
> Сравните, какой вариант лучше извлекает численные значения из таблицы 2 и не теряет их при привязке evidence.

### Пример плохого `creator_prompt`
> Посмотрите вообще на всё: и на temporal scope, и на figures, и на качество hypothesis, и на новизну идеи.

Почему плохо:
- слишком много задач сразу;
- эксперт-участник не понимает главный критерий.

### Как писать `creator_rationale`
Это объяснение, почему кейс диагностический.

Пример:
> Важный факт находится в figure/table, baseline часто теряет численные значения и ошибается в привязке evidence к правильному объекту.

---

## Шаг 6. Заполните блок «C. На что смотреть при ревью»
Здесь есть два набора чекбоксов:
- `review_focus`
- `expected_error_modes`

### Как выбирать `review_focus`
Отмечайте только главное.
Обычно достаточно 1–3 пунктов.

Примеры:
- `evidence`
- `visual_fact`
- `temporal`
- `overall`

### Как выбирать `expected_error_modes`
Отмечайте только те ошибки, которые этот кейс действительно должен уметь вскрывать.

Примеры:
- `missed_visual_fact`
- `wrong_evidence_linkage`
- `needs_time_fix`
- `hallucinated_visual_inference`

### Частая ошибка
Не нужно ставить все чекбоксы подряд.
Если отмечено всё, кейс становится менее диагностическим.

---

## Шаг 7. Используйте блок «D. Техническое сопоставление (advanced)» только при необходимости
Этот блок нужен, если вы хотите точнее привязать кейс к downstream candidate/hypothesis.

Заполняйте только тогда, когда это реально помогает.

Обычно достаточно 1–2 полей:
- `candidate_signature`
- `hypothesis_title_contains`
- `premise_contains`
- `time_scope_contains`

### Когда advanced-блок можно не трогать
- если кейс и так однозначно понятен;
- если manifest создаётся в первую очередь для ручной экспертной проверки;
- если вы пока собираете только черновой набор.

---

## Шаг 8. Пользуйтесь статусом готовности кейса
У каждого кейса есть статус:
- **Готов к экспорту**
- **Нужно дополнить**
- **Не готов**

### Что это значит
Форма автоматически проверяет, хватает ли ключевых полей для нормального использования кейса.

### Что делать, если кейс не готов
Обычно нужно проверить:
- есть ли `paper_title`;
- заполнен ли `creator_prompt`;
- заполнен ли `creator_rationale`;
- выбран ли `review_focus`;
- заполнено ли хотя бы одно `match.*` поле, если это требуется вашим правилам.

---

## Шаг 9. Сохраните черновик, если не закончили
Если вы хотите прерваться:
1. Нажмите **«Скачать черновик JSON»**.
2. Сохраните файл локально.

Когда вернётесь:
1. Откройте форму.
2. Нажмите **«Загрузить черновик JSON»**.
3. Выберите сохранённый файл.

После этого форма восстановит данные.

---

## Шаг 10. При необходимости очистите локальный черновик
Кнопка **«Очистить локальный черновик»** удаляет сохранённую в браузере локальную копию.

Это полезно, если:
- хотите начать заново;
- тестировали форму на демо-данных;
- не хотите, чтобы старые данные автоматически подхватились позже.

Важно:
эта кнопка очищает именно локальное сохранение в браузере.
Текущие данные на экране после нажатия не исчезают автоматически, пока вы не обновите страницу.

## Шаг 11. Скачайте итоговый JSON
Когда кейсы готовы:
1. Нажмите **«Скачать заполненный JSON»**.
2. Сохраните файл `task3_ab_case_manifest.filled.json`.

Это основной результат работы создателя набора.

## Финальный чек-лист перед отправкой

Проверьте, что:
- все нужные кейсы добавлены;
- `easy_control` не доминируют над hard subset;
- у каждого важного кейса есть понятный `creator_prompt`;
- `creator_rationale` объясняет диагностическую ценность;
- отмечены только релевантные `review_focus`;
- отмечены только ожидаемые `expected_error_modes`;
- черновик при необходимости сохранён отдельно;
- итоговый JSON скачан;
- при необходимости данные отправлены через Google Форму.

Хороший результат — это не просто заполненный JSON, а такой набор кейсов, который:
- помогает увидеть реальную разницу между baseline VLM и SFT/DPO VLM;

- понятен участнику A/B без дополнительных устных пояснений;
- не перегружен лишними или размытыми кейсами.
